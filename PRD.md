# 才高题自来 PRD

## 1. 项目背景

本项目来源于 AI 提示词工程师岗位测试题：指定一个应用名，使用 AI 完成一款面向移动端、具备完整功能和一定审美的 H5 产品 Demo，并保留 AI 完成过程截图或录屏，以及最终产品 Demo 的截图或录屏。

选择应用名：才高题自来。

产品方向：面向求职者、学生和学习者的 AI 题目生成与训练工具。用户可以根据岗位、考试方向或知识点快速生成定制题单，并完成答题、评分、复盘和错题积累。

一句话定位：

> 输入目标，题目自来；练完之后，知道下一步该补什么。

## 2. 产品目标

### 2.1 核心目标

打造一个可在移动端演示的 H5 Demo，完整展示以下能力：

- 将模糊应用名转化为清晰产品方案。
- 使用 AI 辅助完成产品规划、界面设计、文案和前端实现。
- 构建完整用户路径：选择目标、生成题单、答题训练、查看评分、沉淀错题。
- 体现提示词工程师能力：通过结构化 Prompt 模拟 AI 出题、AI 追问、AI 评分和复习建议。
- 满足应用商店工具类产品的基本形态，而不是单页展示页面。

### 2.2 演示目标

演示时应能完成一条连续路径：

1. 用户进入首页。
2. 选择或输入训练目标，例如“AI 提示词工程师面试”。
3. 设置题目类型、难度和数量。
4. 点击生成题单，看到 AI 生成过程。
5. 进入答题训练。
6. 完成答题后查看得分、解析、追问建议和复习计划。
7. 在历史记录或错题本中查看本次练习沉淀。

## 3. 目标用户

### 3.1 核心用户

- 准备面试的求职者。
- 正在学习某个知识点、但不知道如何开始练习的人。
- 需要快速生成题目或训练材料的学生。
- 想模拟岗位面试、提升回答质量的转行者。

### 3.2 本次 Demo 的重点用户场景

以“AI 提示词工程师求职者”为重点场景：

- 用户需要了解岗位常见问题。
- 用户需要练习场景题和开放题。
- 用户需要知道自己的回答哪里不完整。
- 用户需要获得下一轮训练建议。

## 4. 用户需求

### 4.1 基础需求

- 我想快速生成一组和目标岗位相关的题目。
- 我想选择题目难度，避免题目太简单或太难。
- 我想用手机直接练习，不需要安装 App。
- 我想答完后看到评分、解析和改进建议。
- 我想保存历史练习记录，方便回顾。
- 我想把答错或回答不好的题目放进错题本。

### 4.2 进阶需求

- 我想让系统根据我的回答继续追问，模拟真实面试。
- 我想知道自己薄弱的能力维度，例如需求理解、Prompt 结构化、风险边界、结果评估。
- 我想一键生成下一组专项练习。
- 我想看到“AI 正在思考/生成”的过程，让产品更有智能感。

## 5. 产品用途

### 5.1 对真实用户的用途

- 面试前快速刷题。
- 针对某个岗位进行模拟训练。
- 针对薄弱知识点生成专项题目。
- 记录练习过程，形成复习闭环。

### 5.2 对本次 HR 测试的用途

- 展示 AI 产品思维：从名字到产品定位、用户路径和功能闭环。
- 展示提示词能力：通过 AI 出题、追问、评分、建议体现 Prompt 设计能力。
- 展示前端交付能力：做出移动端可交互 Demo。
- 展示审美和体验：页面不是粗糙表单，而是有产品完成度。
- 展示工程边界意识：前端不暴露真实模型 API Key，后续可扩展后端代理。

## 6. 产品范围

### 6.1 MVP 必做功能

#### 首页

- 产品名称和今日训练状态。
- 快速训练入口。
- 推荐训练方向，例如 AI 提示词工程师、前端开发、产品经理、运营面试。
- 今日练习概览：已练题数、正确率、待复习错题。

#### AI 生成题单

- 选择训练方向。
- 选择难度：入门、进阶、面试冲刺。
- 选择题型：单选题、问答题、场景题、混合题。
- 选择题目数量。
- 点击生成后展示 AI 生成过程。
- 生成结果进入题单预览。

#### 答题训练

- 一题一屏，适配移动端。
- 支持单选题选择答案。
- 支持开放题输入回答。
- 支持查看提示。
- 支持收藏题目。
- 支持下一题/上一题。

#### 结果分析

- 展示总分和完成情况。
- 展示正确题、薄弱项、错题。
- 针对开放题给出模拟 AI 点评。
- 给出下一步复习建议。
- 支持进入错题本或再来一组。

#### 历史记录与错题本

- 保存生成过的题单。
- 保存练习结果。
- 保存错题和收藏题。
- 支持再次练习。

### 6.2 加分功能

- 追问模式：根据用户回答展示下一轮追问。
- 语音读题：使用浏览器原生 SpeechSynthesis 朗读题目。
- 本地持久化：使用 localStorage 或 IndexedDB 保存历史数据。
- 移动端适配：iPhone、Android 主流宽度适配。
- 可测试链接：部署到 Vercel、Netlify 或 GitHub Pages。

### 6.3 暂不做功能

- 用户登录注册。
- 真实支付。
- 多用户云端同步。
- 真实数据库后台管理。
- 真正接入大模型 API。
- 原生 App 打包上架。

这些能力不作为本次 Demo 的核心范围，因为题目要求是 H5 产品 Demo，重点在产品完整度、AI 使用过程和可演示体验。

## 7. 权限边界

### 7.1 前端权限

本 Demo 默认仅使用浏览器本地能力：

- localStorage：保存历史练习、错题、收藏题。
- SpeechSynthesis：可选，用于语音读题。
- 页面路由：在 H5 内部切换页面。

### 7.2 不申请的权限

- 不申请摄像头权限。
- 不申请通讯录权限。
- 不申请定位权限。
- 不申请短信权限。
- 不申请相册权限。
- 不申请麦克风权限，除非后续明确增加语音答题功能。

### 7.3 数据边界

- Demo 数据默认存储在用户本地浏览器。
- 不上传用户输入内容。
- 不采集个人隐私信息。
- 不在前端写入任何真实 AI API Key。

### 7.4 AI 能力边界

MVP 阶段采用 Mock AI，即通过预设题库、规则组合和模拟生成动画实现 AI 产品体验。

如后续接入真实大模型，应采用后端代理：

- 前端只请求自有后端接口。
- API Key 放在后端环境变量。
- 后端负责模型调用、内容审核、频率限制和日志脱敏。
- 前端不直接调用 OpenAI、DeepSeek、通义等模型服务。

## 8. 技术栈选型

### 8.1 推荐技术栈

- 前端框架：React
- 开发语言：TypeScript
- 构建工具：Vite
- 样式方案：CSS Modules 或原生 CSS 变量加模块化 CSS
- 图标库：lucide-react
- 路由：React Router
- 本地存储：localStorage，必要时扩展 IndexedDB
- 部署：Vercel、Netlify 或 GitHub Pages

### 8.2 选型理由

#### React

- 组件化清晰，适合拆分首页、生成页、答题页、结果页、错题本等模块。
- 生态成熟，容易快速完成 Demo。
- AI 生成 React 代码的稳定性较高。

#### TypeScript

- 适合定义题目、选项、用户答案、练习结果等数据结构。
- 减少后期联调和状态管理错误。
- 面试展示时更显工程规范。

#### Vite

- 启动快，配置轻。
- 适合短时间完成高质量 H5 Demo。
- 构建产物适合静态部署。

#### CSS 变量和模块化样式

- 能做出更有辨识度的界面，不依赖重型 UI 组件库。
- 方便统一主题色、字号、间距和移动端适配。
- 比直接套模板更能体现设计能力。

#### localStorage

- 满足本次 Demo 的历史记录、错题本和收藏需求。
- 不需要真实后端即可形成数据闭环。
- 降低部署和演示风险。

### 8.3 为什么不优先做真实后端和数据库

本次题目重点是 H5 产品 Demo，而不是完整商业系统。真实后端和数据库会增加以下成本：

- 后端接口设计和部署。
- 数据库连接、表结构和迁移。
- 环境变量配置。
- 跨域处理。
- 真实 AI API Key 安全问题。
- 网络不稳定导致现场演示失败。

因此 MVP 推荐纯前端实现，但代码结构预留后端扩展接口。

## 9. 信息架构

底部导航建议包含 4 个主入口：

- 首页
- 生成
- 练习
- 我的

页面结构：

- 首页：训练概览、推荐题单、快速入口。
- 生成页：选择方向、难度、题型、数量并生成题单。
- 题单页：预览题目列表。
- 答题页：逐题答题。
- 结果页：评分、解析、复习建议。
- 我的页：历史记录、错题本、收藏题。

## 10. 核心业务流程

### 10.1 生成题单流程

1. 用户选择训练方向。
2. 用户选择难度、题型和数量。
3. 系统展示生成动画和生成步骤。
4. 系统从题库中组合题目。
5. 系统生成题单摘要。
6. 用户进入练习。

### 10.2 答题流程

1. 用户进入第一题。
2. 系统展示题干、选项或输入框。
3. 用户作答。
4. 用户进入下一题。
5. 完成全部题目。
6. 系统计算结果并生成复盘。

### 10.3 复盘流程

1. 系统展示总分。
2. 系统列出正确题和错题。
3. 系统根据错题标签生成薄弱项。
4. 系统给出复习建议。
5. 用户可保存错题或再次训练。

## 11. 数据模型

### 11.1 Question

- id：题目 ID
- title：题干
- type：题型，single 或 open
- options：选项列表
- answer：标准答案
- explanation：解析
- tags：能力标签
- difficulty：难度
- followUp：追问问题

### 11.2 PracticeSession

- id：练习 ID
- topic：训练方向
- difficulty：难度
- questionType：题型
- questions：题目列表
- answers：用户答案
- score：得分
- createdAt：创建时间
- completedAt：完成时间

### 11.3 UserProgress

- totalPracticed：累计练习题数
- accuracy：正确率
- weakTags：薄弱能力标签
- favorites：收藏题目
- mistakes：错题列表

## 12. 视觉与交互方向

### 12.1 设计关键词

- 聪明
- 清爽
- 学习感
- 工具感
- 移动端友好

### 12.2 推荐风格

采用“纸质练习册 + 智能工具”的视觉方向：

- 背景：暖白或浅纸色。
- 主色：墨绿色，体现理性和学习感。
- 强调色：明亮黄色或荧光绿，用于生成按钮、分数和重点状态。
- 文字：高对比、易阅读。
- 动效：生成题单时使用步骤流转和轻量进度动效。

### 12.3 移动端适配

- 以 375px 宽度作为主要设计基准。
- 兼容 320px 到 430px 常见手机宽度。
- 使用安全区适配底部导航。
- 按钮高度不低于 44px。
- 表单控件和答题选项保持足够点击面积。

## 13. 技术难点

### 13.1 AI 体验真实性

难点：纯前端 Demo 如果只是显示固定题目，容易显得不够智能。

解决方案：

- 增加生成过程动画，例如“分析岗位能力”“匹配题型”“生成追问”“整理评分规则”。
- 题目根据方向、难度、题型动态筛选和组合。
- 结果页根据用户答案和题目标签生成不同复习建议。

### 13.2 开放题评分

难点：没有真实后端 AI 时，开放题无法真正理解用户回答。

解决方案：

- 使用关键词匹配和答案长度进行模拟评分。
- 根据题目标签生成结构化点评。
- 明确这是 Demo 中的模拟 AI 评分，后续可接入真实大模型。

### 13.3 状态流转

难点：生成题单、答题、结果、历史记录之间状态较多，容易混乱。

解决方案：

- 明确定义 PracticeSession 数据结构。
- 统一由当前练习会话驱动页面状态。
- 完成后将结果写入 localStorage。

### 13.4 移动端体验

难点：手机端空间有限，题目、选项、解析和按钮容易拥挤。

解决方案：

- 一题一屏。
- 底部固定主操作按钮。
- 解析和提示采用可展开区域。
- 使用清晰层级，减少同屏信息噪音。

### 13.5 演示稳定性

难点：现场演示时网络、接口、环境变量可能出问题。

解决方案：

- MVP 使用纯前端 Mock 数据。
- 可离线或静态部署访问。
- 真实 AI 接口只作为后续扩展，不影响 Demo 主流程。

## 14. 后续可扩展方向

### 14.1 轻量后端

如时间充足，可增加一个简单后端代理：

- Node.js + Express 或 serverless function。
- 提供 `/api/generate-questions` 和 `/api/evaluate-answer`。
- 接入真实大模型。
- API Key 存放在服务端环境变量。

### 14.2 数据库

如需要用户系统，可增加：

- Supabase：用户登录、练习记录、错题本。
- PostgreSQL：结构化保存题目和练习结果。

但这些不作为本次 Demo 必须内容。

## 15. 验收标准

### 15.1 功能验收

- 能在手机尺寸下正常浏览。
- 能生成题单。
- 能完成答题。
- 能展示结果分析。
- 能保存历史记录。
- 能查看错题或收藏。
- 刷新页面后关键数据仍存在。

### 15.2 体验验收

- 首屏能明确看出产品名称和用途。
- 页面视觉完成度高，不像默认模板。
- 交互路径顺畅，无明显断点。
- 按钮、选项、输入框在移动端容易点击。
- 文字不溢出、不遮挡、不拥挤。

### 15.3 交付验收

- 提供最终 Demo 截图或录屏。
- 提供 AI 使用过程截图或录屏。
- 提供可本地运行或线上访问的 Demo。
- 提供项目说明文档。

## 16. 推荐开发顺序

1. 搭建 React + Vite + TypeScript 项目。
2. 定义题目、练习会话和用户进度数据结构。
3. 完成首页和底部导航。
4. 完成生成题单页。
5. 完成答题页。
6. 完成结果页。
7. 完成历史记录和错题本。
8. 加入 localStorage 持久化。
9. 做移动端视觉优化。
10. 录制 AI 使用过程和产品演示过程。

## 17. 最终建议

本项目应优先追求“完整、稳定、好演示”，而不是堆叠复杂技术。

最优方案是：

> 纯前端移动端 H5 + 本地持久化 + Mock AI 生成/评分 + 可部署链接 + 完整过程录屏。

如仍有额外时间，再增加后端代理真实 AI 接口，而不是一开始就把项目复杂化。
